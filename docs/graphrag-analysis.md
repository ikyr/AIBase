# GraphRAG 引入可行性分析

## 1. 当前检索能力盘点

项目已有一套可用的混合检索系统，知识表示以**扁平文本块（chunk）** 为单位，存储在 PostgreSQL + Milvus 中。

| 能力 | 实现 | 说明 |
|---|---|---|
| 向量搜索 | Milvus (IVF_FLAT + COSINE) | 查询嵌入 → topK*2 候选 |
| 关键词搜索 | SQL ILIKE | 短语匹配 0.8 / 单词匹配 0.5 启发式打分 |
| 融合排序 | RRF (Reciprocal Rank Fusion) | k=60，向量 + 关键词结果合并后取 topK |
| 双存储后端 | Milvus / RAGFlow | 通过 `knowledge.repository.type` 配置切换 |
| Agent 检索 | ReAct 循环中调用 `knowledge_search` 工具 | searchType: HYBRID |
| 上下文管理 | 固定窗口 (8000 token) + 后向裁剪 | 旧消息截断为 200 字符摘要 |

**当前缺失：**

- 无实体提取、无关系建模、无本体
- 无图数据库（整个代码库零 Neo4j/SPARQL/RDF 引用）
- 无基于图的重新排序或上下文扩展
- "GraphBridge" 连接的是 DAG 工作流引擎，非知识图谱

---

## 2. GraphRAG 核心能力

GraphRAG（Microsoft）不是向量搜索的替代品，而是**补充**。其核心流程：

```
文档 → LLM 实体/关系提取 → 知识图谱构建 → 社区检测 → 社区摘要生成
                                                          ↓
用户查询 ← 图遍历 + 社区摘要 + 向量检索 ← 多路融合 ← 查询入口
```

三个关键差异化能力：

| 能力 | 传统 RAG | GraphRAG |
|---|---|---|
| 局部检索（"X 是什么"） | Chunk 向量匹配 | Chunk 向量匹配 |
| 全局理解（"这个数据集整体在说什么"） | 难以回答 | 社区摘要直接给出 |
| 多跳推理（"A 和 B 之间有什么关联"） | 依赖运气，chunk 之间无连接 | 沿实体-关系路径遍历 |

---

## 3. 典型适用场景

- **跨文档综合分析**：上传 50 份财报，问"这三家公司过去五年在 AI 领域的投资策略有什么异同？"——纯向量搜索很难答好，GraphRAG 的社区摘要和实体遍历能给出结构化答案。

- **事件脉络梳理**：多份文档中提到同一事件的不同侧面，GraphRAG 可以沿实体关系串联出完整时间线。

- **知识发现**：用户不知道自己要找什么时，社区摘要提供"这个知识库覆盖了哪些主题"的宏观视角。

---

## 4. 引入代价

| 维度 | 代价 |
|---|---|
| 索引成本 | 每个文档需多次 LLM 调用（实体提取 + 关系提取 + 社区摘要），成本远超普通 embedding |
| 基础设施 | 需引入图数据库（Neo4j），增加运维复杂度 |
| 索引时效 | 文档增删改后需重建图索引，更新延迟高于向量库 |
| 工程复杂度 | 新增实体/关系提取管道、图存储层、图谱检索分支、多路融合逻辑 |
| 调优门槛 | 实体提取粒度、关系类型设计、社区检测参数均需针对业务场景调优 |

---

## 5. 判断结论

**当前阶段不建议引入 GraphRAG。** 理由：

1. **项目处于早期**。基础 CRUD、ReAct Agent、工作流引擎仍在搭建，当前混合检索已能满足"查找包含某关键词的文档"类查询。

2. **收益取决于场景成熟度**。如果用户的主要问题是单点查找，向量 + 关键词足够；当实际出现大量**跨文档综合分析**需求时，GraphRAG 的投入才值得。

3. **先验证再建设**。用现有系统收集用户反馈，观察检索失败模式。如果反复出现"找不到"或"答案不完整"且根因是跨文档关联而非 chunk 召回，才是引入 GraphRAG 的时机。

---

## 6. 引入条件（触发点）

满足以下 **2 项及以上** 时，重新评估：

- [ ] 知识库文档量级达到数千份，且主题分散
- [ ] 用户频繁提出跨文档对比、总结、关联类问题
- [ ] 现有检索系统反馈"找不到"或"答案不完整"的比例超过阈值（如 15%）
- [ ] 有明确的多跳推理需求（问题需要串联 3 个以上 chunk 才能回答）

---

## 7. 未来引入技术路线（规划稿）

若条件成熟决定引入，建议以下架构：

```
                        ┌─────────────┐
                        │  查询入口    │
                        └──────┬──────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                 ▼
        ┌──────────┐    ┌──────────┐     ┌──────────┐
        │ 向量检索  │    │ 关键词检索 │    │ 图谱检索  │  ← 新增分支
        │ (Milvus) │    │ (ILIKE)  │     │ (Neo4j)  │
        └────┬─────┘    └────┬─────┘     └────┬─────┘
             │               │                │
             └───────────────┼────────────────┘
                             ▼
                    ┌────────────────┐
                    │   三路 RRF 融合  │  ← 向量 + 关键词 + 图遍历
                    └────────────────┘
```

**分阶段实施：**

| 阶段 | 内容 | 产出 |
|---|---|---|
| Phase 1 | Neo4j 部署 + 实体/关系 schema 设计 | 图存储就绪 |
| Phase 2 | Ingest Pipeline 增加实体/关系提取步骤 | 文档入库时自动建图 |
| Phase 3 | 图谱检索分支实现 + 三路 RRF 融合 | 查询侧透明升级 |
| Phase 4 | 社区检测 + 社区摘要生成 | 全局问答能力 |

**对 Agent 透明：** `knowledge_search` 工具接口保持不变，返回结果质量提升，Agent 无需感知底层检索机制变化。
